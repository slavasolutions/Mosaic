The spec

Three rules. Nothing more.

Every concept in the spec — sidecars, identity, cascade, refs — derives from one of these three. If a rule isn't here, it isn't Mosaic.

RULE 01

A file is a record.

.json files hold structure. .md, .pdf, .png — opaque payloads. Structured metadata for any file goes in a .json sidecar with the same name.

# structured about.json → record # opaque + sidecar hello.md hello.json → sidecar
RULE 02

A folder is a collection.

Any directory is a collection. Folders nest to any depth. index.json (or index.md) inside a folder is that folder's record.

# team/ is a collection team/ index.json → team record ada.json → team/ada grace.md grace.json → team/grace
RULE 03

The filename is the contract.

Names follow name[.modifier]*.ext. Identity is form-independent — file form and folder form are the same record. No rename, no broken refs.

# same identity: about about.json about/index.json → identical # variant modifier about.fr.md → locale fr

How it works

Reading a record is one pass.

The pipeline is fixed, single-pass, never re-entered. This predictability is what keeps Mosaic from growing an evaluator.

STEP 1
Content
.json object, or empty object for opaque files
↓
+
STEP 2
Sidecar merge
Sidecar JSON merged on top. Sidecar wins on collision. Shallow.
↓
+
STEP 3
Cascade fill
Fills absent keys from ancestor defaults. Only locale by default; profiles declare more.
↓
→
STEP 4
Refs resolved
ref: strings replaced with their targets. Pure: same folder → same result, always.
No evaluator. No recursion. No XPath.

References

Link records with ref:

A reference is a JSON string starting with ref:. Two anchor forms, one optional JSON Pointer. That's the entire grammar.

"ref:team/ada"
Absolute. Resolved from the root, regardless of where the reference appears. Recommended default.
"ref:./other"
Relative. Resolved against the referring record's folder. Use for intra-collection links that should survive the folder being moved.
"ref:team/ada#/email"
JSON Pointer. RFC 6901 path after # selects a value inside the resolved target. Yields the string "ada@x.io".

No wildcards. No predicates. No content-queries. The ceiling against re-deriving XPath is permanent and explicit in the spec.

Scope

What Mosaic is not.

Pre-empting the most common misconceptions, because a clear boundary is part of the spec.

  • ✕
    Not a CMS No admin UI, no auth, no media pipeline. Mosaic is a folder shape, not a product.
  • ✕
    Not a database No query language, no indexes, no transactions. The filesystem is the store; a reader is not a DB engine.
  • ✕
    Not a build tool Mosaic doesn't build anything. Astro, Next, a custom Rust script — they build. Mosaic is what they read.
  • ✕
    Not a runtime No daemon, no server, no required process. A conforming folder is readable with a filesystem and a JSON parser.
  • ✕
    Not an opinion about templates No layout system, no component format, no design tokens (yet). Engines decide how content is rendered.
  • ✕
    Not an opinion about design No theming, no color tokens, no breakpoint conventions. That's the engine's domain, not the format's.
It's a folder shape. Tools layer on top. The spec is the contract between the folder and the tools.
